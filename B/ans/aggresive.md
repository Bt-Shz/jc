The ACM Code of Ethics is useful because it starts from a simple but often ignored point: computing is not neutral. A software system can affect privacy, safety, access to opportunities, and public trust. The lecture on the ACM Code explains that computing professionals should contribute to society, avoid harm, respect privacy, and act fairly (Lecture, Feb. 6). I agree with those values, but the Code still feels stronger as a statement of ideals than as a tool for holding people accountable. I would revise it so that its principles require clearer professional practices: life-cycle impact audits, measurable fairness testing, and stronger user control over data and agency. Without those kinds of requirements, the Code risks sounding morally serious while remaining too easy to ignore.

The first change should be a direct requirement for life-cycle impact audits. The Code already tells professionals to evaluate systems and risks, but that duty should be more specific (Lecture, Feb. 6). Before a system is released, teams should write down the likely harms, the groups most likely to be affected, possible failure modes, non-AI fallback options, and ways for users to report or correct problems. After release, they should keep checking whether the system is causing harm that was not predicted. This is especially important for AI and machine learning systems because uncertainty is part of how these systems work. The machine learning fairness lecture warned that unknown-unknown errors and false high confidence are dangerous because users may not realize when a system is wrong or know how to challenge it (Lecture, Jan. 23). So “avoid harm” should not just mean having good intentions before launch. It should mean proving that risks were reviewed before, during, and after deployment.

The second change should make fairness and bias testing more measurable. The ACM Code is right to say that computing professionals should act fairly, but “do not discriminate” is too broad by itself. A team should have to explain how it tested for discrimination, not only claim that it cares about fairness. Bias can come from imbalanced datasets, correlated variables, lurking features, or even from the way the problem is defined (Lecture, Jan. 23). Removing race or gender from a model does not automatically make the model fair, because other variables, such as location, income, or education history, can work as proxies. The Code should require professionals to name the fairness standard they are using, explain why it fits the situation, and provide evidence that the system was tested before release. This does not reduce fairness to a technical checklist. Fairness is still social and contextual. But it does make it harder for teams to hide behind vague language. In areas such as hiring, healthcare, facial recognition, and credit scoring, ethical responsibility should include documented bias analysis, not just a general promise to be fair.

The third change should expand privacy into a stronger right to data control and user agency. The privacy lecture explains that privacy is not only about secrecy. It is also about autonomy and self-determination, including control over when and how personal information is shared (Lecture, Mar. 12). Modern systems make this control harder because they can collect, combine, infer, store, and share data in ways users may not understand. Even anonymized data can sometimes be re-identified, and trained models may leak information through memorization or model inversion (Lecture, Mar. 12). The Code should therefore require meaningful user control over what is collected, inferred, remembered, shared, and deleted. This should also apply to manipulative design. In human-AI interaction, users may become attached to AI assistants or companions, while companies may profit from that attachment through longer engagement (Lecture, Feb. 27). Respect for users should include a duty not to exploit confusion, vulnerability, or dependency.

Students should encounter the ACM Code in their first programming or computer science course, before they get used to treating technical skill as separate from social responsibility. But reading it once is not enough. The Code should return in software engineering, data science, AI, capstone projects, internships, and workplace onboarding. It should also be tied to moments when people gain real power, such as accessing user data, deploying public systems, approving model releases, or conducting user studies. Otherwise, ethics becomes something students discuss in class and then quietly leave behind when the real work begins.

Consequences for violating the Code should depend on intent, negligence, and harm. Honest mistakes should lead to correction plans, ethics training, and additional review. Repeated negligence or refusal to assess risks should lead to formal discipline, supervision, or removal from a project. Serious cases, such as intentional exploitation, hidden harms, falsified audits, or misuse of sensitive data, should allow job termination, professional sanctions, and legal referral when needed. The lecture notes that ACM membership can be terminated, but this is limited because many technologists are not ACM members (Lecture, Feb. 6). Real accountability therefore has to come from universities, companies, and professional organizations too.

Overall, the ACM Code should remain a moral foundation for computing, but it should also demand evidence of ethical practice. Public good, fairness, and privacy do not happen just because professionals say they value them. They require review, testing, user control, and real consequences when foreseeable harms are ignored.
