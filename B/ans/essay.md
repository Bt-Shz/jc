The ACM Code of Ethics is valuable because it recognizes that computing is not ethically neutral: technical decisions shape social life, distribute risks, and affect public trust. The lecture on the ACM Code emphasizes that computing professionals should contribute to society, avoid harm, respect privacy, and act fairly (Lecture, Feb. 6). However, the Code is stronger as a statement of values than as a guide for accountability. I would revise it so that its principles require concrete professional practices: lifecycle impact audits, measurable fairness testing, and stronger user control over data and agency. These changes would make the Code less aspirational and more useful for real decisions in computing work.

The first change should be an explicit requirement for lifecycle impact audits. The Code already expects professionals to evaluate systems and risks, but this duty should be stated more operationally (Lecture, Feb. 6). Before deployment, teams should document likely harms, affected groups, failure modes, non-AI fallback options, and procedures for user feedback or correction. After deployment, they should monitor whether the system is causing unexpected harm and update it when problems appear. This is especially important for AI and machine learning systems, where uncertainty is not a minor detail but part of the technology itself. The machine learning fairness lecture warned that unknown-unknown errors and false high confidence are dangerous because users may not realize when a system is wrong or know how to challenge it (Lecture, Jan. 23). For that reason, “avoid harm” should not only mean trying to build safe systems. It should mean showing that risks were reviewed throughout the system’s life.

The second change should make fairness and bias testing more measurable. The ACM principle of fairness is necessary, but “do not discriminate” is too broad unless professionals must explain how they tested for discrimination. Bias can enter through imbalanced datasets, correlated variables, lurking features, or poorly framed problems (Lecture, Jan. 23). A team might remove race or gender from a model and still reproduce discrimination through proxies such as location, income, or education history. Therefore, the Code should require professionals to identify which fairness standard they are using, justify why it fits the context, and provide evidence that the system was tested before launch. This would not make fairness purely technical, since fairness is also social and cultural, but it would stop teams from treating fairness as a slogan. In areas such as hiring, healthcare, facial recognition, and credit scoring, ethical responsibility should include documented bias analysis, not only good intentions.

The third change should strengthen privacy into a broader right to data control and user agency. The privacy lecture explains that privacy is not just secrecy; it is connected to autonomy and self-determination, including control over when and how personal information is shared (Lecture, Mar. 12). Modern computing systems make this more difficult because they can collect, aggregate, infer, retain, and share information in ways users may not understand. Even anonymized data can sometimes be re-identified, and trained models may leak information through memorization or model inversion (Lecture, Mar. 12). The Code should therefore require meaningful user control over what is collected, inferred, remembered, shared, and deleted. This duty should also cover manipulative design. In human-AI interaction, users may become dependent on AI assistants or companions, while companies may benefit from emotional attachment and prolonged engagement (Lecture, Feb. 27). Respecting users should include avoiding designs that exploit confusion, vulnerability, or dependency.

Technologists should first encounter the Code in their first programming or computer science course, before they learn to treat technical skill as separate from social responsibility. But the Code should not be a one-time reading assignment. It should be reintroduced in software engineering, data science, AI, capstone projects, internships, and workplace onboarding. Agreement to the Code should be connected to practical authority: accessing user data, deploying public systems, approving model releases, or conducting user studies. In this way, ethics becomes part of professional practice rather than a symbolic checkbox.

Consequences for violating the Code should be proportional to intent, negligence, and harm. Honest mistakes should lead to correction plans, ethics training, and additional review. Repeated negligence or refusal to assess risks should lead to formal discipline, supervision, or removal from a project. Serious cases, such as intentional exploitation, concealed harms, falsified audits, or misuse of sensitive data, should allow job termination, professional sanctions, and legal referral where appropriate. The lecture notes that ACM membership termination is possible, but this is limited because many technologists are not ACM members (Lecture, Feb. 6). Real accountability must therefore come from universities, companies, and professional organizations.

Overall, the ACM Code should remain a moral foundation for computing, but it should also require evidence of ethical practice. Public good, fairness, and privacy are not achieved simply by naming them. They require review, testing, user control, and consequences when professionals ignore foreseeable harm.
