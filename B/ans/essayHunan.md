The ACM Code of Ethics matters because computing is never just technical. Software systems affect people’s privacy, safety, opportunities, and trust in institutions. The lecture on the ACM Code explains that computing professionals should contribute to society, avoid harm, respect privacy, and act fairly (Lecture, Feb. 6). I agree with those values, but I think the Code is still stronger as a moral statement than as a practical guide for accountability. I would revise it by adding clearer professional requirements: life-cycle impact audits, measurable fairness testing, and stronger user control over data and agency. These changes would make the Code easier to apply in real computing work.

The first change should be an explicit requirement for life-cycle impact audits. The Code already expects professionals to evaluate systems and risks, but that responsibility should be more concrete (Lecture, Feb. 6). Before deployment, teams should document likely harms, affected groups, possible failure modes, non-AI fallback options, and procedures for user feedback or correction. After deployment, they should keep checking whether the system is causing unexpected harm and update it when problems appear. This matters especially for AI and machine learning systems, where uncertainty is built into the technology. The machine learning fairness lecture warned that unknown-unknown errors and false high confidence can be dangerous because users may not know when a system is wrong or how to challenge it (Lecture, Jan. 23). For that reason, "avoid harm" should mean more than trying to build a safe system. It should require professionals to show that risks were reviewed throughout the system’s life.

The second change should make fairness and bias testing more measurable. The ACM principle of fairness is necessary, but "do not discriminate" is too broad unless professionals also have to explain how they tested for discrimination. Bias can come from imbalanced datasets, correlated variables, lurking features, or the way a problem is framed in the first place (Lecture, Jan. 23). A team might remove race or gender from a model and still reproduce discrimination through proxies such as location, income, or education history. The Code should therefore require professionals to identify the fairness standard they are using, explain why that standard fits the context, and provide evidence that the system was tested before launch. This would not make fairness purely technical, because fairness is also social and cultural. But it would make it harder for teams to treat fairness as a vague promise. In areas such as hiring, healthcare, facial recognition, and credit scoring, ethical responsibility should include documented bias analysis, not just good intentions.

The third change should expand privacy into a broader right to data control and user agency. The privacy lecture explains that privacy is not only secrecy. It is also connected to autonomy and self-determination, including control over when and how personal information is shared (Lecture, Mar. 12). Modern computing systems make this harder because they can collect, combine, infer, store, and share information in ways users may not understand. Even anonymized data can sometimes be re-identified, and trained models may leak information through memorization or model inversion (Lecture, Mar. 12). The Code should require meaningful user control over what is collected, inferred, remembered, shared, and deleted. This responsibility should also include manipulative design. In human-AI interaction, users may become dependent on AI assistants or companions, while companies may benefit from emotional attachment and longer engagement (Lecture, Feb. 27). Respecting users should include avoiding designs that exploit confusion, vulnerability, or dependency.

Students should first encounter the ACM Code in their first programming or computer science course, before they start seeing technical skill as separate from social responsibility. Still, the Code should not be treated as a one-time reading. It should return in software engineering, data science, AI, capstone projects, internships, and workplace onboarding. Agreement to the Code should also be connected to real authority, such as accessing user data, deploying public systems, approving model releases, or conducting user studies. That would make ethics part of professional practice instead of a symbolic checkbox.

Consequences for violating the Code should depend on intent, negligence, and harm. Honest mistakes should lead to correction plans, ethics training, and further review. Repeated negligence or refusal to assess risks should lead to formal discipline, supervision, or removal from a project. Serious cases, such as intentional exploitation, concealed harms, falsified audits, or misuse of sensitive data, should allow job termination, professional sanctions, and legal referral when appropriate. The lecture notes that ACM membership termination is possible, but this has limited force because many technologists are not ACM members (Lecture, Feb. 6). Real accountability therefore has to come from universities, companies, and professional organizations too.

Overall, the ACM Code should remain a moral foundation for computing, but it should also require evidence of ethical practice. Public good, fairness, and privacy are not achieved by naming them. They require review, testing, user control, and consequences when professionals ignore foreseeable harm.
